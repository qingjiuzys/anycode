执行摘要

一句话结论：内部系统接口不稳定、文档缺失时，客户被迫走浏览器自动化；但浏览器方案慢、贵、易卡住。应建立「Python API 抓取 → 结构化表格输出 → 浏览器仅兜底」的分层策略，并同步优化审批策略与 Token 预算控制。

问题逐条详解

每条包含：客户原话要点、业务影响、根因分析、建议应对。

1

抓取表格数据

客户需要从 ERP / OA / 报表系统中导出或复制结构化表格，供后续分析或填报。

中
客户诉求
稳定拿到「行 × 列」结构化数据，字段对齐、可复用，不要截图或手工复制。
典型场景
销售明细表、库存台账、审批流水、对账单等后台列表页。
痛点
页面分页、懒加载、嵌套 iframe、导出按钮权限不一；纯 LLM「看页面」无法保证表格完整性。
建议方案
  • 优先：抓包 / 文档还原 API，Python 脚本拉取 JSON 后转 CSV（见 #6）。
  • 次选：DOM 选择器定点抽取表格 <table> 或虚拟列表 API。
  • 兜底：浏览器全页截图 + OCR（仅无接口且无稳定 DOM 时）。
2

系统不确定性高，需浏览器方式集成内部系统；但浏览器较慢

内部系统无开放 API、版本迭代快、前端框架各异，只能模拟真人操作浏览器。

高
客户诉求
在「不能改后端、不能接 SSO 统一网关」的前提下，仍要自动化登录、点选、填表、翻页。
根因
  • 遗留系统：接口未文档化，或有接口但鉴权复杂。
  • 前端 SPA：元素定位随发版变化，维护成本高。
  • 浏览器链路：启动 → 登录 → 等待渲染 → 逐步操作，端到端延迟显著高于 API。
业务影响
单次任务从秒级变成分钟级；批量跑批在上班时间会占满客户端资源；客户对「实时性」预期落空。
建议方案
  • 建立连接器分层：L1 官方/逆向 API → L2 轻量 HTTP → L3 无头浏览器。
  • 浏览器会话复用登录态（Cookie / Storage），避免每次冷启动。
  • 对确定性步骤脚本化，仅把「异常分支」交给 LLM 决策，降低不确定性带来的重试。
  • 售前话术：明确「浏览器是兼容性方案，不是性能方案」。
权衡说明：浏览器集成覆盖面最广，但是当前客户抱怨「慢」的第一来源。必须在方案评审阶段就标注预期耗时，避免交付后口碑反转。
3

频繁确认（审批打断）

Agent 每执行一步敏感操作都弹审批，长任务被切成大量碎片，用户体验差。

中
客户诉求
批量、重复、可预期的操作希望「一次授权、全程执行」；只有真正高风险动作才打断。
关联成本
每次确认 = 一次等待 + 可能的多轮 LLM 重规划 → 直接推高 Token 消耗（见 #5）。
建议方案
  • 任务级白名单：对指定项目 / 脚本 / 工具链开启「本会话自动批准」。
  • 风险分级：读操作默认放行；写操作 / 外发 / 支付类才拦截。
  • 批处理模式：预演（dry-run）一次 → 用户确认计划 → 无人值守执行。
  • 微信 / IM 遥控：支持异步审批，避免人守在电脑前点确认。
4

Node Playwright 脚本运行速度很慢

客户侧或竞品方案常用 Node + Playwright，实测端到端耗时难以接受。

高
慢在哪里
  • Node 冷启动 + Playwright 拉起 Chromium（数百毫秒～数秒）。
  • 默认 waitForTimeout / 网络空闲等待过于保守。
  • 逐步截图、逐步 dump DOM 给 LLM，IO 与序列化开销大。
  • 单线程逐步执行，未并行化独立子任务。
对比认知
客户常把「Playwright 慢」等同于「AI 自动化慢」，影响产品选型。
建议方案
  • 能不用 Playwright 就不用；必须用时常驻浏览器进程 + CDP 直连。
  • 操作脚本与 LLM 解耦：脚本负责 80% 确定性步骤，LLM 只处理异常。
  • 减少全页截图频率，改用元素级快照或 accessibility tree。
  • 评估 Rust / Python 侧轻量驱动，或复用已有登录会话的 Browser MCP。
5

Token 成本太高

长上下文、多轮工具调用、页面内容灌入模型，导致单次任务费用远超客户心理价位。

高
成本来源
  • 浏览器 DOM / 截图 base64 进入上下文。
  • 表格全量回传而非摘要或分页。
  • 失败后反复重试，历史消息累积。
  • 频繁确认导致任务被拆成更多 turn（见 #3）。
客户预期
希望「跑一次几毛钱」，实际可能「跑一次几块钱甚至更高」。
建议方案
  • 上下文裁剪：工具结果只保留结构化摘要；大表落盘传路径。
  • 模型分级：规划用大模型，抽取/分类用小模型。
  • 预算门禁：任务级 Token 上限 + 超预算自动降级或中止。
  • 可观测：Dashboard 展示单次任务 Token / 费用，售前可量化报价。
商务风险：若不提前展示 Token 曲线，客户会把成本问题归咎于「产品不好用」而非「用法不当」。
6

使用 Python 进行 API 层抓取

客户与技术团队更倾向于 Python 脚本直连 HTTP API，而非浏览器模拟。

机会点
为什么倾向 Python
  • 企业内数据团队熟悉 Pandas / Requests，易维护。
  • 抓包逆向后可直接复用 Cookie / Token，毫秒～秒级响应。
  • 输出标准 CSV / Parquet，方便进 BI 或二次计算。
与 #1 #2 的关系
表格抓取（#1）的最佳实践；可规避浏览器慢（#2）与 Playwright 慢（#4）的主要部分。
建议方案
  • 提供 API 连接器模板（Python）：鉴权、分页、字段映射、落盘。
  • Agent 调用方式：Bash 执行脚本 → 只把摘要表头 + 行数 + 文件路径回传 LLM。
  • 建立内部「接口资产库」：每个客户系统一条连接器记录，版本化管理。
  • 仅当 API 失效时自动降级到浏览器，并告警维护连接器。
7

复杂任务较慢；无自行计算字段时 WorkBuddy 容易卡住

任务步骤多、但缺少需要推理/计算的中间字段时，编排引擎空转或长时间无进展。

高
客户原话要点
「明明只是按流程点一遍，为什么要思考那么久？」「跑到一半就不动了。」
典型卡住场景
  • 纯搬运型流程：登录 → 打开菜单 → 查询 → 导出，无衍生指标计算。
  • 页面元素未出现 / 选择器失效，Agent 反复「思考」却无有效工具调用。
  • 多系统串行，前一步未校验成功就进入下一步，导致死等。
根因分析
  • 编排粒度不当：该脚本化的流程仍交给 LLM 逐步即兴决策。
  • 缺少进度心跳：用户看不到当前步骤，误以为卡死。
  • 无超时 / 无降级：单步等待无上限，失败不自动切换 API 路径。
  • 竞品对标：WorkBuddy 在「无计算字段」的线性任务上表现不佳，客户有直观对比。
建议方案
  • 引入 Workflow / Runbook：线性任务用 YAML 步骤机执行，LLM 仅负责填槽位。
  • 每步设置 超时 + 重试 + 失败分支（如切换 API、通知人工）。
  • Dashboard / 微信推送步骤级进度（received → running step 3/7 → done）。
  • 对「无计算字段」任务做快速通道：检测到纯搬运模式则跳过深度推理模型。

客户会议话术参考

可直接引用或改写，用于管理预期、引导技术选型。

关于浏览器方案

「浏览器自动化是我们兼容遗留系统的兜底能力，不是默认路径。它的优势是覆盖面，代价是速度和维护成本。如果能在两周内逆向出 API，后续每次运行会从分钟级降到秒级。」

关于 Token 成本

「成本主要来自把整页内容反复交给模型。我们会把表格落盘、只传摘要，并对纯脚本步骤不走大模型，这样单次任务费用可下降一个数量级。」

关于卡住 / WorkBuddy 对比

「没有计算步骤的线性流程，应该用固定 Runbook 执行而不是让 AI 逐步即兴想。我们会加步骤进度和超时，您在微信上能看到『第 3/7 步』而不是长时间无响应。」
