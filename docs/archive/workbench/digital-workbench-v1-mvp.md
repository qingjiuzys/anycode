# Digital Workbench V1 MVP

> **Status (2026-05):** M1–M5, WebUI MVP, and **V2 slices A–D** are **complete**. User guide: [docs-site/guide/dashboard.md](../docs-site/guide/dashboard.md). Planning: [digital-workbench-next-steps.md](digital-workbench-next-steps.md). V3+: see [digital-workbench-handoff.md](digital-workbench-handoff.md).

## Goal

Ship a local, project-level dashboard that makes anycode usable as a reliable AI automation system for real projects.

V1 is not a generic enterprise platform yet. V1 must prove one loop:

```text
project -> session/run -> structured events -> SQLite -> WebUI -> gates/artifacts/trust
```

## Product Scope

### In Scope

1. **Project registry**
   - Detect/open a project by workspace root.
   - Persist project metadata in `~/.anycode/projects.db`.
   - A project owns sessions, events, gates, artifacts, and automation policies.

2. **Session/run ingestion**
   - Every `anycode run`, `goal`, `workflow`, and stream REPL session creates a project session.
   - Existing `output.log` remains compatible.
   - New structured events are appended to SQLite.

3. **Dashboard WebUI**
   - Command: `anycode dashboard`.
   - Local server only: default `127.0.0.1`.
   - Pages:
     - Login/local trusted entry
     - Home dashboard
     - Projects
     - Project conversations/sessions
     - Automations
     - Assets
     - Agent/Skills
     - Settings

4. **Gates**
   - Gate results are first-class records.
   - Gate examples:
     - `cargo fmt`
     - `cargo clippy`
     - `cargo test`
     - `flutter analyze`
     - `flutter test`
     - README marker
     - browser/manual verification
   - A task cannot be “trusted complete” if required gates failed.

5. **Artifacts**
   - Track files and reports touched or generated by agents.
   - Track provenance:
     - project
     - session
     - agent
     - gate verification

6. **Settings**
   - SQLite database path.
   - Dashboard host/port.
   - Auth mode.
   - Model fallback policy.
   - Asset read policy.
   - Skills enablement.

### Out of Scope for V1

- Remote SaaS deployment.
- Team collaboration over network.
- Full SSO/OIDC implementation.
- Tauri/native desktop wrapper.
- Full connector system for GitHub/Linear/Slack/Sentry.
- Bidirectional task control from UI, except safe read-only actions.

## V1 UX Acceptance

1. `anycode dashboard` opens or prints a local URL.
2. User can select a project.
3. User can see recent sessions under that project.
4. User can inspect a running goal:
   - attempts
   - active agent
   - last error
   - latest tool calls
   - gate status
5. User can see why a task is not trusted complete.
6. User can see project artifacts and which gate verified them.
7. User can configure dashboard port and SQLite path.

## V1 Engineering Acceptance

1. SQLite database created at `~/.anycode/projects.db`.
2. Schema migrations are versioned.
3. `RunLogger` or equivalent writes structured events.
4. Legacy `output.log` remains unchanged.
5. Dashboard API reads from SQLite.
6. SSE streams new project events.
7. WebUI is built from React/Vite and embedded or served by CLI.
8. Rust tests cover:
   - migration
   - event insert
   - project/session association
   - gate failure blocks trusted completion
   - API serialization

## Milestones

### M1: Data Foundation

- Add SQLite dependency.
- Add `projects.db` migration runner.
- Implement `projects`, `sessions`, `project_events`, `gates`, `artifacts`.
- Add unit tests.

### M2: Runtime Event Sink

- Extend runtime logging to also emit structured events.
- Preserve `output.log`.
- Associate events with project/session/task.

### M3: Dashboard API

- Add local server:
  - `/api/projects`
  - `/api/projects/:id`
  - `/api/projects/:id/sessions`
  - `/api/sessions/:id/events`
  - `/api/projects/:id/gates`
  - `/api/projects/:id/artifacts`
  - `/api/events/stream`

### M4: WebUI

- Implement React app from `docs/digital-workbench-app`.
- Pages:
  - home
  - projects
  - conversations/sessions
  - automations
  - assets
  - agents/skills
  - settings

### M5: Gate Trust Model

- Add `trusted_status` to sessions/artifacts.
- Mark session as:
  - `running`
  - `completed_unverified`
  - `trusted_complete`
  - `failed`
  - `blocked`

## First Implementation Slice

The first slice should be small:

1. Create SQLite schema.
2. Insert project/session rows for `anycode run`.
3. Insert task start/end and tool call events.
4. Serve `/api/projects` and `/api/projects/:id/sessions`.
5. Show those rows in WebUI Projects page.

No auth, no control actions, no remote deployment in the first slice.
